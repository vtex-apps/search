# AGENTS.md — search

## Repository Purpose

`vtex.search` is a VTEX IO **Store Framework app** (builders: `react`, `store`, `messages`, `styles`, `docs`) that exports the **search-companion blocks** for VTEX storefronts: autocomplete result list (v2), did-you-mean correction, search suggestions, and search banner. It is a sibling of `vtex.search-result` — it does **not** implement any layer of the four-layer Intelligent Search flow (session → schema → resolver → API → IS) and does **not** call `vtex.intelligent-search-api` directly.

It also exports a small client utility (`BiggyClient`) that storefront and host apps use to issue Apollo queries against `vtex.store-resources` (autocomplete suggestions, top searches, suggestion products).

## Where it sits in the IS topology

| App | Role |
|---|---|
| `vtex.search-session` | Owns the `search` vtex.session namespace |
| `vtex.search-graphql` | App-owned GraphQL schema |
| `vtex.search-resolver` | GraphQL resolvers |
| `vtex.intelligent-search-api` | Backend HTTP wrapper around the IS platform |
| `vtex.search-result` | Store Framework PLP blocks |
| **`vtex.search`** (this repo) | Storefront search-companion blocks (autocomplete v2, did-you-mean, suggestions, banner) |
| `vtex.delivery-promise-components` | Shopper-location blocks |

This repo lives **next to** `vtex.search-result`, not above or below it. Both are storefront consumers reading from the same GraphQL surface (via `vtex.store-resources`).

## Architecture

```
Storefront route → header / search input
    │
    └─► <autocomplete-result-list.v2>  (Autocomplete.js)
            │
            ├─ BiggyClient (react/utils/biggy-client.ts)
            │     ├─ topSearches            (vtex.store-resources/QueryTopSearches)
            │     ├─ suggestionSearches     (vtex.store-resources/QueryAutocompleteSearchSuggestions)
            │     └─ suggestionProducts     (vtex.store-resources/QuerySuggestionProducts)
            │           variables include cookies('sp-variant') + advertisementOptions
            │
            ├─ <product-summary>            (vtex.product-summary — required child)
            ├─ <search-suggestions>         (Suggestions.js — text-side suggestions)
            └─ history cookie               cookie name 'biggy-search-history' (limit 5)

Search results page (PLP)
    │
    ├─► <did-you-mean>                     (DidYouMean.js — correction prompt)
    └─► <search-banner>                    (Banner.js — IS-driven banner slot)
```

`BiggyClient` is the only place this repo issues GraphQL queries. It reads/writes two cookies:

- `biggy-search-history` — local search history for the autocomplete dropdown (max 5 entries).
- `sp-variant` — read-only here; passed through as the `variant` query variable so the IS backend can route to the active A/B experiment.

This app does **not** read or write `localStorage['vtex.search.pickupInPoint']` (owned by `vtex.delivery-promise-components`) and does **not** issue requests against `/_v/api/intelligent-search/*` (owned by `vtex.intelligent-search-api`).

## Autonomy Limits

### Toolbelt and platform
- **Never** run `vtex link`, `vtex publish`, `vtex deploy`, workspace promotion, or any account/workspace-changing Toolbelt command without explicit confirmation.
- **Ask before** running `vtex setup --tooling` — this repo uses `eslint-config-vtex@^12` + `eslint-config-vtex-react@^6` with custom rules; `vtex setup --tooling` could overwrite `.eslintrc`/`.prettierrc`.
- **Never** modify `manifest.json` `version` directly — use `vtex release <patch|minor|major> stable`. The `prereleasy` script (`bash pre.sh`) runs as part of the release flow.
- **Never** modify `react/typings/` — regenerated by `vtex setup --typings --ignore-linked`.
- **Ask before** adding/removing `builders` or `policies` in `manifest.json`. The `policies` array is currently empty; introducing outbound policies must be reviewed by the platform team.

### Block contract
- **Never** remove a block name from `store/interfaces.json` without a coordinated deprecation cycle — themes in production rely on `autocomplete-result-list.v2`, `did-you-mean`, `search-suggestions`, and `search-banner` being stable.
- `autocomplete-result-list.v2` requires `product-summary` (`vtex.product-summary`). Removing that requirement is breaking for any theme relying on the default composition.
- The `.v2` suffix on `autocomplete-result-list` is intentional. There is no `.v1` exported here today; do not invent one without coordinated PRs in storefront themes.

### Cookies and storage
- **`biggy-search-history`** cookie is **owned by this repo**. Format is a comma-joined list capped at 5 entries (see `BiggyClient.prependSearchHistory`). Changing the format breaks history rendering on legacy carts.
- **`sp-variant`** cookie is **read-only** here — the IS A/B routing system writes it elsewhere. Reading it for the `variant` GraphQL variable is the only intended use.
- This app does **not** read `localStorage['vtex.search.pickupInPoint']` and **does not** participate in pickup-point session updates. If autocomplete behavior must change based on pickup point, route through `vtex.search-result` / `vtex.delivery-promise-components`, not from here.

### GraphQL queries
- All queries are imported from `vtex.store-resources` (e.g. `vtex.store-resources/QueryAutocompleteSearchSuggestions`). **Never** hand-write GraphQL strings against the IS API or `vtex.search-graphql` directly — that bypasses caching and shopping-segment gating provided by `store-resources`.
- The `BiggyClient.suggestionProducts` signature accepts many optional params (`hideUnavailableItems`, `orderBy`, `count`, `shippingOptions`, `advertisementOptions`, …). Adding params is safe; removing or reordering is a breaking change for host apps importing `BiggyClient`.

### Branch / release flow
- Current default branch is **`master`** (not yet migrated to `main`). PR workflow triggers on `master`. Confirm before any branch rename.
- Prereleasy hook: `bash pre.sh`. Inspect it before changing publish behavior.

### Test maintenance
- No Jest/Mocha tests exist today. Pre-PR validation is lint-only locally, plus Cypress in CI (against `biggy`). Adding a Jest setup is a deliberate engineering decision — coordinate with the team before introducing one (it requires a Jest config, vtex-test-tools install, and possibly a coverage threshold to align with sibling repos).

### Legacy `node/` folder
- `node/package.json` and `node/yarn.lock` exist but **no `node` builder is declared in `manifest.json`**. Toolbelt does not package or link this folder. Treat it as dead. Do not extend it; removing it should be its own PR after confirming nothing in CI or release tooling references it.
